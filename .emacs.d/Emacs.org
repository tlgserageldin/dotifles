* Programming Support

** LSP Mode

Language Server Protocol client for Emacs. Provides IDE-like features:
- Go-to-definition (`M-.`)
- Find references
- Hover docs
- Code actions / refactoring
- Symbol rename
- Completions via company/capf
- Per-language intelligence via external LSP servers

This setup loads `lsp-mode` lazily and activates LSP in common programming modes
using `lsp-deferred` to avoid blocking startup.

#+begin_src emacs-lisp
  (use-package lsp-mode
    :commands (lsp lsp-deferred)
    :init
    ;; Prefix for lsp-command-map (C-c l ...)
    (setq lsp-keymap-prefix "C-c l")

    ;; Performance / UX settings
    (setq lsp-enable-symbol-highlighting t
          lsp-enable-snippet t
          lsp-prefer-flymake nil) ;; prefer flycheck if present

    :hook ((c-mode          . lsp-deferred)
           (c++-mode        . lsp-deferred)
           (python-mode     . lsp-deferred)
           (rust-mode       . lsp-deferred)
           (go-mode         . lsp-deferred)
           (js-mode         . lsp-deferred)
           (js-ts-mode      . lsp-deferred)
           (typescript-mode . lsp-deferred)
           (tsx-ts-mode     . lsp-deferred)
           (lua-mode        . lsp-deferred)
           (sh-mode         . lsp-deferred)))

  ;; UI extras (hover docs, sideline info)
  (use-package lsp-ui
    :after lsp-mode
    :commands lsp-ui-mode)

  ;; Company for completions (via capf)
  (use-package company
    :config
    (global-company-mode 1))
  
  (with-eval-after-load 'company
    (add-to-list 'company-backends 'company-capf))
#+end_src

*** Verifying LSP Setup
After installing servers:

1. Open a file in the relevant mode (e.g. =.c=, =.py=, =.rs=).
2. Ensure =lsp-mode= starts (look for “LSP” in the modeline).
3. Test:
   - =M-.= → go to definition
   - =M-?= → find references
   - =C-c l= → LSP command map
** LSP Debugging
Useful tools and workflows when an LSP server is not starting or not behaving as expected.

*** Check LSP Logs
Shows communication between Emacs and the language server.

#+begin_src emacs-list :tangle no
;; Open the main LSP log buffer
(lsp-workspace-show-log)
#+end_src

Or interactively:

- =M-x lsp-describe-session=
- =M-x lsp-workspace-show-log=

These buffers often reveal missing executables, crashed servers, or invalid workspace roots.

*** Restart the LSP Server
Sometimes the server needs a clean restart after config or dependency changes.

#+begin_src emacs-list :tangle no
(lsp-workspace-restart)
#+end_src

Interactive:

- =M-x lsp-workspace-restart=
- =C-c l r= (if using lsp-command-map)

*** Verify the Server Executable
If the server isn’t starting, confirm it's on PATH. Examples:

#+begin_src shell
which clangd
which pyright
which rust-analyzer
which gopls
#+end_src

If Emacs was launched from GUI (macOS), PATH may differ from shell PATH.  
Solution: initialize PATH inside Emacs:

#+begin_src emacs-list :tangle no
(use-package exec-path-from-shell
  :config
  (exec-path-from-shell-initialize))
#+end_src

*** Check the Project Root Detection
LSP uses project “root markers” (e.g., =compile_commands.json=, =pyproject.toml=, =go.mod=, =Cargo.toml=).

If detection fails, override manually:

#+begin_src emacs-list :tangle no
;; Force a directory as LSP project root
(setq lsp-client-packages '(lsp-clients))
#+end_src

Use:

- =M-x lsp-workspace-folders-add=
- =M-x lsp-workspace-folders-remove=

*** Common Diagnoses

- **Server installed but not found** → PATH issue  
- **Server exits immediately** → missing dependency, invalid config file  
- **Definition jumps don't work** → incomplete project root  
- **Hover/docs not showing** → missing capabilities in server or disabled in lsp-ui  
- **High CPU** → disable semantic tokens or code lenses  

#+begin_src emacs-list :tangle no
(setq lsp-semantic-tokens-enable nil)
(setq lsp-lens-enable nil)
#+end_src

*** Check Capabilities
Useful to see what features the server supports:

- =M-x lsp-capabilities=

*** Enable Verbose LSP Logs (for deep debugging)
#+begin_src emacs-list :tangle no
(setq lsp-log-io t)
#+end_src

This logs every message sent/received.  
Turn it off afterward; it gets noisy.

*** Final Step: Clean Workspace
Sometimes leftover metadata causes odd behavior.

- =M-x lsp-workspace-restart=
- or delete =.lsp-session-v1= file in =~/.emacs.d/=

* Version Control
** Magit
Magit is an interactive, porcelain-style Git interface for Emacs.  
Key features:
- Status buffer (`magit-status`) as the main entry point  
- Stage/unstage hunks or lines  
- Commit, amend, reword, squash, fixup  
- Browse logs, branches, remotes, stashes  
- Interactive rebase UI  
- Resolve merge conflicts in-buffer  
- Push/pull with popups guiding available actions  
- Transient-based menus allow discoverability (`?` always shows help)

Key bindings (once Magit loads):
- =C-x g= → `magit-status` for the current project  
- =s= / =u= → stage / unstage  
- =c c= → commit  
- =P u= → push to upstream  
- =F u= → pull from upstream  
- =b b= → switch branches  
- =l l= → log view  
- =TAB= → expand a section  
- =RET= → act on thing at point  

Configuration below adds Magit, enables autosaving before commits, and improves commit message editing.

#+begin_src emacs-lisp
(use-package magit
  :commands (magit-status magit-blame)
  :bind (("C-x g" . magit-status))
  :config
  ;; Save buffers automatically when invoking Magit commands
  (setq magit-save-repository-buffers 'dontask)

  ;; Use the full frame for commit messages (optional)
  (setq git-commit-fill-column 72
        git-commit-summary-max-length 50)

  ;; Show word-level diffs by default
  (setq magit-diff-refine-hunk 'all))
#+end_src

** Magit Notes
- Magit creates a “status buffer” representing repository state, not a shell interface.  
- Everything is interactive: press ? on any popup to see available actions.  
- Magit uses Transient menus: press key prefixes, then subcommands (e.g. =c c=).  
- Rebase workflows are guided via `magit-rebase`; Magit validates operations before executing them.  
- The “diff → stage → commit → push” loop mirrors Git’s conceptual model but exposes more structure.  
- Magit automatically detects project roots (via Git repos) and uses them as context for all actions.
